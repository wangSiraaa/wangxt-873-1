# 问询坐席分流调度台 (Call Center Dispatch Console)

基于 React 18 + TypeScript 5 + Vite 6 + TailwindCSS 3 + Zustand 5 实现的**本地模拟版**问询坐席分流调度台，数据全部本地持久化（localStorage），刷新页面后队列、倒计时、转派记录全部保留。

---

## ✨ 功能特性

### 📋 核心数据
| 数据类别 | 内容 |
|---------|------|
| 问题类型 | 业务咨询 / 投诉处理 / 技术支持 / 账单查询 / 其他 |
| 技能标签 | 业务知识、产品知识、沟通技巧、投诉处理、技术知识、故障排查、财务知识、账单系统、通用服务、英语能力 |
| 坐席状态 | 空闲 / 忙碌 / 暂停 / 离线 |
| 客户等级 | 新客户→普通→银牌→金牌→钻石VIP（5星级） |
| 转派原因 | 6项预设 + 自由填写，5字符强制校验 |
| 升级记录 | 保留来源(超时/手动)、升级时间、等待秒数、描述 |

### 👥 角色权限
| 操作 | 班长 (Monitor) | 普通坐席 (Agent) |
|------|:-:|:-:|
| 分派候选池号码 | ✅ | ❌ |
| 分派升级队列 | ✅ | ❌ |
| 转派所有处理中号码 | ✅ | 仅限本人处理的 |
| 完成所有处理中号码 | ✅ | 仅限本人处理的 |
| 导出当班记录 | ✅ | ❌ |
| 修改他人坐席状态 | ✅ | ❌ |
| 修改本人坐席状态 | ✅ | ✅ |

### 🔄 主流程
1. **候选池进入** → VIP/黑名单置顶 → 星级降序 → 入池时间升序
2. **自动推荐坐席** → 技能匹配×30 + 负载空余×10 + 空闲奖励20，取TOP3
3. **手动分派** → 弹窗显示智能推荐，离线/暂停自动禁用
4. **处理中转派** → 必须填写转派原因(≥5字符)，保留前后坐席与操作员
5. **暂停接单** → 坐席点击切换状态，暂停坐席所有分派被拦截
6. **超时升级** → 等待≥300s自动移入升级队列，带倒计时来源与Toast提醒
7. **处理完成** → 处理结果 + 处理状态(已完成/已转出) + 备注，归档到已完成
8. **导出当班** → 班长一键导出UTF-8 BOM格式CSV，Excel中文兼容

### 🔒 状态规则（可见反馈）
- ⛔ **离线或暂停坐席不能接单**：分派/转派下拉选项自动禁用+标红，按钮层二次拦截，Store层三次校验
- ⏱ **等待超时自动升级**：倒计时<30s橙色预警，超时后移入红色升级队列并保留来源说明
- 📝 **转派必须填写原因**：<5字符按钮禁用，失焦红框+错误文字，提交再校验
- 👤 **普通坐席不能修改别人状态**：切换他人状态弹出错误Toast，AgentList他人卡片灰显
- 📊 **黑名单/高优影响排序**：高优紫色标识→黑名单红色标识→普通排序，**但不绕过离线拦截**

### 💾 本地持久化
持久化键：`call_center_dispatch_v1`（白名单，不持久化弹窗和Toast）：
- 角色 / 当前坐席身份 / 超时阈值
- 坐席列表（含状态、负载）
- 4个队列（全部保留，Date 通过 reviver + onRehydrateStorage 双保险恢复）
- **倒计时存绝对时间戳而非秒数**，刷新后基于Date.now()重算，无缝衔接

## 🚀 快速开始

### 方式一：本地开发（推荐）
```bash
# 1. 安装依赖
npm install

# 2. 启动开发服务（默认 http://localhost:5173）
npm run dev

# 3. 生产构建
npm run build
npm run preview
```

### 方式二：Docker 容器启动
```bash
# 1. 构建镜像
docker build -t call-center-dispatch:latest .

# 2. 启动容器（端口映射 8080→80）
docker run --rm -p 8080:80 --name dispatch-console call-center-dispatch:latest

# 3. 浏览器打开
open http://localhost:8080
```

## ✅ 验收路径（3条强制必过）

### 🧪 方式 A：浏览器手动验收
启动后按以下顺序操作：

#### 验收路径 1：转派号码但不填原因无法提交
1. 默认是**班长**视角，处理中队列有3个号码；
2. 点击任一中号码的「转派」按钮打开弹窗；
3. **目标坐席**随便选一个空闲的（如李娜）；
4. **转派原因**留空或只输入 "1234" （少于5字符）；
5. ✅ **确认**：
   - 「确认转派」按钮呈灰色禁用状态；
   - 如果点击原因输入框后失焦，下方出现红色错误提示："请填写转派原因" 或 "转派原因至少需要5个字符"；
   - 即使绕过按钮禁用，提交时弹出错误 Toast，号码也不会被转派。

#### 验收路径 2：尝试分派给离线坐席被阻止
1. 候选排队池第一个号码是 138****8888（钻石VIP），点击「分派坐席」；
2. 弹出分派弹窗，在**全部坐席**下拉框中找到 "陈静"（离线）和 "刘芳"（暂停）；
3. ✅ **确认**：
   - 陈静 / 刘芳 的 option 是灰色禁用，并在后面标有 ⛔ 该坐席当前离线 / 已暂停；
   - **智能推荐 TOP3** 列表里完全没有陈静和刘芳（离线/暂停直接排除）；
   - 即使在下拉强行选中（浏览器调试修改disabled），点击分派也会弹出错误 Toast："无法分派：该坐席当前离线，无法接单"，号码仍留在候选池。

#### 验收路径 3：刷新后已升级号码仍在升级队列
1. 观察**升级队列**面板，初始应存在号码 `133****7777`（投诉超时-黄先生）；
2. 点击该号码卡片上的红色边框确认来源是「候选池等待超时5分钟未分派」；
3. 浏览器手动刷新（⌘R / Ctrl+R）或点击顶栏的「重置数据」→取消再刷新；
4. ✅ **确认**：
   - 刷新后**升级队列仍包含 133****7777**；
   - 升级时间、超时描述、红色来源标签全部原样保留；
   - 倒计时基于升级记录的 `upgradeAt` 绝对时间戳重新计算，不发生跳变。

### 🧪 方式 B：一键运行验收脚本（需浏览器已打开页面）
```bash
# 运行类型检查 + 生产构建（已自动包含上述 3 条路径的断言式代码分析）
bash ./scripts/verify.sh
```
脚本将：
1. 运行 `tsc` 类型检查；
2. 运行 `vite build` 构建产物；
3. 校验关键文件中的 3 条业务规则代码断言（离线拦截、转派校验、升级持久化）；
4. 输出最终 PASS / FAIL 报告。
